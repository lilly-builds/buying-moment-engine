## D9 safety audit (the reason this gate exists)

**Sends never fire at a real practice, PASS, proven three independent ways.**
- Two fail-closed gates, both required, both run **before any network I/O** (`src/send/guard.ts` `assertSandboxTarget`): (a) caller-supplied `classification === "sandbox"`, and (b) the address is on the `SEND_SANDBOX_*` env allowlist. Re-checked a second time in `writeAndEnroll` (`src/send/hubspot-send.ts:831`).
- Classification is derived from the practice's own `geoKey.startsWith("demo:")` (`src/send/send-brief.ts:441`). Prod has **0** `demo:` practices → every real practice is `real_practice` → blocked regardless of address. And no `SEND_SANDBOX_*` env is set → the address check fails for everyone too.
- **Network-spy proof (project's own tests):** `tests/send/hubspot-send.test.ts:218` "BLOCKS a real-practice recipient and makes ZERO network calls (spy)" → `expect(calls).toHaveLength(0)`; plus the smuggled-sandbox-address twin. **`vitest run tests/send tests/outreach` → 57 passed.**
- **Third UI-level gate:** every live brief currently has no sendable contact email → the brief shows "No contact email on this brief yet, so there is nothing to send." (no fireable Send control at all).
- **Outreach adapter gated off:** `src/outreach/gate.ts`, no `OUTREACH_*` env → `state = awaiting_credentials`, message "ready, awaiting Outreach credentials". Inert.

**No patient/PHI-shaped fields anywhere, PASS.** Full schema read (`db/schema/*`): every table `.enableRLS()`; `contacts` is explicitly business-only with a nullable `name` (honest role-only fallback). Grep over `db/schema` + `db/migrations/*.sql` for patient/PHI column names → the only hits are the "ZERO PHI / no patient-shaped fields" guarantee comments. In `src/`/`app/`, "patient" appears only in (a) the LLM guardrail prompt ("Never report information about patients. Business information only.", `src/enrich/research-prompt.ts:21`) and (b) marketing pack copy. No PHI storage field exists.

**No keys or contact data in logs, PASS.** Grep of every `console.*`/`logger` in `src/`+`app/`+`jobs/` for interpolated email/token/secret/key/contact/credential → **zero hits.** The two riskiest spots are hardened: the OAuth error logs only `errorType`, deliberately NOT `Error.message` ("Drizzle can include bound params… do not log the message here", `src/crm/oauth-flow.ts:89`); detector fetch-failure logs carry `{source, query, error.message}` and Google Places logs only `{placeId, practiceHint}`, never the key-bearing request URL. `src/crm/token-crypto.ts` never logs; provider keys stored as ciphertext, decrypted server-side only, never returned to client.

## Findings (severity-ranked)

**F1, Mobile: the app is desktop-first and overflows on phones. [medium · polish, not safety]**
Confirmed in DevTools device mode (iPhone 14 Pro Max, 430px): content squished into a narrow left column with a white gutter; nav runs off-screen. Root cause: three shared components hardcode a single horizontal row with no `sm:`/`md:` responsive variants, `design/components/top-nav.tsx` (nav, no wrap/menu), `design/components/section-header.tsx` (`flex-row justify-between` + `shrink-0` action at all widths), `design/components/segmented-control.tsx` (`whitespace-nowrap` pills, no wrap/scroll). Reused on feed, brief, scoreboard, integrations → whole app affected. **Handed off:** `eliseai/thread-prompts/16-mobile-responsive-design.md` (fix on a branch → PR to main, Lilly merges).

**F2, Only 1 of 3 detectors is producing signals. [SHIP BLOCKER, Lilly's call 2026-07-11]**
All 26 signals are `phone_complaints` from the Google Places discovery path (`signal_source = discovery:eliseai`). The whole product thesis is multi-signal buying moments ranked by count (D3/D8); with only one signal type, the demo cannot show its core value, so this blocks ship-readiness (not a safety issue, but a demo-credibility one). **Root cause (evidence):** `raw_signals` table is EMPTY (0 rows), so the detector registry produced nothing; `cost_events` show `adzuna`=1 (one test call) and `gdelt`=0, so the Adzuna + GDELT detectors effectively never ran. The registry (`jobs/run-detectors.ts`) is code-complete but not wired into the populate/cron flow actually being used. **Handed off:** `eliseai/thread-prompts/17-light-up-all-three-detectors.md` (Part A: run the detectors at all; Part B: stack via Thread 08's cross-check + name-matcher fix).

**F3, HubSpot one-for-one match is unproven (0 pushes). [low · unverified checklist item]**
No lead has been pushed (`crm_links=0`), so U15's "HubSpot records match the feed one-for-one" can't be checked. A connection exists (`crm_connections=1`). To close it: push one sandbox/real lead through and diff the tags (vertical · signal-source · signal-count · quality).

**F4, Scoreboard is honest but empty. [low · expected]**
`roi_events=0` (correctly wiped of demo data), so every measured/modeled tile shows 0 or a no-data dash. By design (only real AE activity counts). It will stay empty until real meetings/feedback happen, so the ROI story isn't demonstrable on live data yet.

**F5, Observed a 2-touch sequence in the UI; spec calls for a 3-touch sequence. [low · verify]**
Bone Drs brief rendered "First email" + "Follow-up 2" (2 touches). Touch copy lives in `briefs.voice` (the `sequences` table is empty). Verify whether briefs consistently produce 3 touches or the brief-yield fix (#22) trims thin practices to 2.
