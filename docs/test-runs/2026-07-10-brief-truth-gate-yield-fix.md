# Case Study — "The brief writer is hallucinating stats" (2026-07-10)

**One line:** 2 of 3 real briefs were being rejected by the anti-hallucination gate — but it
*wasn't* hallucination. The model was borrowing **real** numbers from the sales playbook into
the wrong part of the email, and the gate correctly refused them. The fix raised yield without
weakening the gate. (Companion to `2026-07-10-first-real-pipeline-run.md`, which surfaced the
symptom. Shipped in PR #22.)

## Background — what the gate is and why it exists

Every AE-facing brief is generated by an LLM (Opus 4.8) and then forced through three gates
before it can be saved: **shape** (valid JSON), **closure** (every cited evidence id was really
in the input), and **truth** (`lint.ts` — no number appears in the prose that isn't in the
evidence). The truth gate is the product's whole trust promise: a rep reads this brief 30 seconds
before dialing, and a fabricated "you're missing 40% of your calls" is exactly the kind of thing a
prospect corrects them on. So the gate is deliberately unforgiving, and a brief that fails it is
**not saved** — the practice simply gets no brief.

The good news from Test Run #1: the gate works. Nothing false ships. The bad news: it was killing
briefs we actually wanted, and nobody yet knew *why*.

## The investigation — how we found the cause without guessing

The trap here is to "fix" it from a hunch. The run log only recorded violation *counts*, not the
offending number, so the first discipline was: **don't touch code until we've seen the actual
rejected text and the specific token.**

1. **Reconstructed the "legal numbers" deterministically.** The gate's allowed-number set is built
   in code from the practice's evidence + the vertical pack — no LLM needed. Rebuilding it for each
   failing practice showed the decisive fact: **their evidence-number set was empty `[]`.** These
   practices had literally no number of their own that the brief was allowed to state anywhere
   except inside a rebuttal.
2. **Re-ran the writer with a token dump** (read-only, no DB writes) to capture the *actual*
   rejected number and the field it was in. Every failure was a **real pack number in an email
   touch**: `5` from "the first five minutes" (copied from the pack's example opener), `3` from
   "a three-location eye-care group" (the case study).
3. **Ran a positive control.** In the *same* failing brief, the model had used the pack's proof
   numbers **inside the rebuttals** — "700+ calls a month," "50 hours," "200+ appointments,"
   "around 23%" — and the gate **passed every one of them.** Across 8 sampled rejections, not one
   violation ever landed on a rebuttal field.

That control is what turned a hunch into a diagnosis.

## The three hypotheses, and the answer

The open question was framed as three possibilities:

- **(a) The checker is stricter than its own rules and deleting true facts.** → **Ruled out.** The
  positive control proves pack numbers pass cleanly where the rules allow them (rebuttals). The
  checker is not stricter than its prompt.
- **(b) The model fabricated a number and got correctly caught.** → **This is it — with a twist.**
  The numbers were *real* (they came from the pack), not invented; the model just put them in a
  field where neither the prompt nor the gate permits them. Correctly caught.
- **(c) A tokenizer false-positive on a number that was really in the evidence.** → **Not possible
  here** — there were no evidence numbers to mis-read. (One narrow edge existed: a "worth fifteen
  minutes?" CTA that dodged the meeting-duration exemption. Real but minor, not the yield story.)

## Root cause, in plain English

The briefs failed **not because the gate is broken and not because the model invented fake stats.**
They failed because these practices had **nothing of their own to be specific with**, while the
prompt pushes hard to *"teach one thing"* and *"be specific"* and hands the model a playbook full of
vivid numbers. With no legal number of its own, the model reached for the playbook's numbers and put
them in the outreach emails — where the gate only allows a playbook number inside a rebuttal. A
**placement collision under thin evidence.** The gate did its job.

## The fix — raise yield without touching the gate

The line we refused to cross: **don't loosen the truth gate.** That would trade the accuracy the
whole product depends on for a bit more yield. Instead we removed the *reasons* the model overreached.

1. **Prompt (`prompts/voice.ts`).**
   - Deleted the instruction that dangled *"a number from the pack's proof"* as a way to teach — it
     was actively inviting the behavior.
   - **Generalized the numbers rule**: previously only the proof/ROI figures were marked
     "rebuttal-only," which left the model free to lift a number from the pack's *pain line* or its
     *example opener*. Now **every** number anywhere in the pack is another practice's and is
     quotable only inside a rebuttal, and the tone-reference example is labeled "rhythm only, never
     reuse its words or numbers."
2. **Retries (`config.ts`).** `VOICE_MAX_ATTEMPTS` **2 → 4** (the first attempt plus 3 retries; each
   retry hands the model the exact edit list). **This is accuracy-neutral:** the truth gate runs on
   *every* attempt, so a brief that lands on the fourth try is exactly as grounded as one that lands
   first. It buys yield, never laxity. Cost is negligible — briefs are generated once at seeding and
   persisted, so the extra Opus call fires only on the briefs still failing.

Notably, we did **not** edit the authored playbook copy — the fix is entirely in the prompt and the
retry budget.

## Verification (measured, not asserted)

Live, on the two practices that actually failed, truth gate only, no DB writes:

| Practice | Before (2 attempts, old prompt) | Prompt fix, 3 attempts | Prompt fix + 4 attempts |
|---|---|---|---|
| Bone Drs (orthopedics) | ~1/3 | 3/4 | **3/4** |
| Austin Retina (ophthalmology) | failing | 3/4 | **4/4** |

The generalized prompt rule did the heavy lifting (ortho 1/3 → 3/4 at just 3 attempts); the extra
retry took ophthalmology to a clean sweep. `npx vitest run tests/brief/` → **162 pass**; typecheck
clean. The unit test's exhaustion assertions were rebound to `VOICE_MAX_ATTEMPTS` so the expected
count follows the config instead of silently drifting on the next change.

## Known residual & follow-up

Orthopedics keeps a ~1/4 residual even at 4 attempts — the genuinely thinnest case (one weak signal,
zero numbers of its own). Brute-forcing more retries is the wrong answer; the **durable fix is
upstream enrichment**: pull one real practice number (locations / providers / years open) so the
model has legal material to be specific with, instead of leaning on the pack. Tracked as a
follow-up, deliberately out of this PR.

## Transferable lessons (the interview version)

- **"Hallucination" was the wrong frame.** The model wasn't inventing facts — it was misplacing real
  ones. Naming the failure precisely changed the entire fix. The first plausible story is often wrong.
- **A positive control beats staring at failures.** Confirming the gate *accepts* the number where
  it's allowed is what ruled out "the validator is broken" and pointed at "the model put it in the
  wrong place."
- **Reconstruct the deterministic half before spending on the stochastic half.** The set of legal
  numbers was computable from code with zero LLM calls; that alone explained most of the mystery
  before a single dollar of re-runs.
- **Fix the cause, not the gate.** The tempting shortcut — loosen the checker so more briefs pass —
  would have quietly reintroduced the exact risk the checker exists to stop. The accuracy contract is
  the product.
