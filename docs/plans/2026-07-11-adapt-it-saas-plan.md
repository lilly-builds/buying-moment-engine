## The one-sentence vision
Any B2B company signs up, an AI agent interviews them for ~2 minutes, and the whole
buying-moment engine **re-skins and re-configures to their business** on the spot — their
brand, their industry, their buying-moment signals, their pitch, their proof — then hands
them a live feed of prospects with ready-to-send briefs written in their voice.

**Design law:** "Don't Make Me Think." One decision per screen. The AI does the heavy
lifting; the human only confirms or nudges. It should feel effortless and a little magical.

---

## The core architectural idea (the smallest correct change)
The engine already resolves its config from four code-level seams (packs, TenantProfile,
TargetConfig, theme tokens). We add ONE new layer that sits **on top** of those seams and
overrides them when an active workspace exists — otherwise the existing EliseAI defaults
stand untouched (existing tests stay green, nothing is ripped out).

```
Active workspace?  ──yes──▶  read brand + pack + signals + pitch + feed  from  workspaces.config (DB JSON)
        │
        └──no───▶  existing EliseAI code defaults (unchanged)
```

**A `workspaces` row is a self-contained tenant.** It holds a JSON config so we do NOT
have to migrate every entity table or add RLS overnight (that is the documented production
step, not tonight's build). The generated sample feed lives inside the workspace record, so
a new tenant's dashboard is fully populated and on-brand **without** live signal detection.

### The workspace config shape (the whole customization surface, one object)
```ts
Workspace {
  id, slug, createdAt,
  brand:   { productName, companyName, primaryColor, accentColor, heroFrom, heroTo,
             logoText | logoDataUrl, fontChoice },
  business:{ oneLiner, whatYouSell, icp, decisionMakerRoles[], geography },
  signals: Signal[]   // { name, kind, why, dataSource, freshnessDays } — THEIR buying moments
  pitch:   { painFit, opener{ leadWith, vocabulary[], tone, exampleOpener },
             discoveryQuestions[], objections[{q,rebuttal}] },
  proof:   ProofPoint[]  // { claim, metric, sourceUrl | tag:"pending" }
  sampleFeed: SampleProspect[]  // { name, headline, signals[], brief{...} } generated at onboarding
}
```
Every field above is (a) generated by the AI onboarding and (b) editable in the
Customization Studio. That is "super easy to customize to a high extent."

## Scope

### IN — build tonight (all e2e-testable in the browser as a user)
1. **`workspaces` table + resolver.** Drizzle schema + additive migration. `resolveActiveWorkspace()`
   returns the DB workspace or the EliseAI default. Active workspace tracked by cookie.
2. **The AI "Adapter" onboarding.** Delightful one-thing-per-screen conversational flow. Real
   Claude calls: interview → propose buying-moment signals for their business → generate the
   full workspace config (brand, pack, signals, pitch, proof) → generate 3 sample prospect
   briefs in their voice. A live "watch it adapt" reveal at the end.
3. **Runtime theming / instant adaptation.** `BrandProvider` injects per-tenant CSS-var
   overrides. Migrate the 4 JS-hex gradients (`healthHero`, `orb`, `brand`, `signalGradients`)
   + `StatRing` to CSS vars so ONE override re-skins 100% of the surface. Product name,
   company, wordmark read from the active workspace.
4. **The adapted dashboard.** Feed + brief + scoreboard render the workspace's sampleFeed and
   config, wearing the tenant brand. Existing EliseAI feed (practices table) still serves the
   default workspace.
5. **The Customization Studio** (`/customize`). Every lever the AI set, human-editable: brand,
   signals, pitch, proof, ICP. Save → re-skins/reconfigures live.
6. **Landing + entry.** A clean "Adapt this engine to your business" front door; restyle the
   off-system login screen onto the design kit.

### OUT — documented as the production roadmap, NOT built tonight
- Full RLS multi-tenant isolation across every entity table (config-level tenancy tonight).
- Live signal detection for arbitrary industries (sample feed generated instead; the detector
  seams already accept injected queries — wiring documented).
- Per-tenant BYOK key vault + per-tenant HubSpot connect (single-instance connect stays).
- Migrating the `vertical` / `signal_kind` pgEnums (workspace signals are free-form JSON,
  decoupled from the healthcare enums).

## Build phases (each a delegated, well-scoped unit)

- **P1 — Data + resolver foundation.** `workspaces` table, Drizzle schema, migration applied to
  Supabase, `resolveActiveWorkspace()` + active-workspace cookie, EliseAI default fallback, seed
  the default workspace from existing TargetConfig/packs. Zod schema for the config. Tests.
- **P2 — Runtime theming engine.** `BrandProvider` + CSS-var override injection; migrate the 4
  gradients + StatRing to CSS vars; product/company/wordmark from active workspace; logo slot in
  TopNav; restyle login onto the kit. Keep `tokens.ts`/`globals.css` parity test green (overrides
  are a runtime layer, never edits to those files).
- **P3 — The AI Adapter onboarding.** The interview UI (one-thing-per-screen, gorgeous) + the
  server route that runs the Claude interview, proposes signals, and emits the validated config +
  3 sample briefs. The "watch it adapt" reveal. This is the showpiece.
- **P4 — Adapted dashboard + Customization Studio.** Workspace-aware feed/brief rendering from
  sampleFeed; `/customize` editor for every lever with live re-skin.
- **P5 — Landing/entry polish + the Steve-Jobs visual pass.** Front door, surface differentiation
  so screens don't all wear the identical hero, elevation/materiality on cards, motion, empty
  states. Ruthless detail.
